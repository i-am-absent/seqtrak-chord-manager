# Code Pack Sharing Backend Foundation Design

## Goal

Build the Supabase foundation for anonymous code-pack sharing while keeping the static React frontend independent from the backend implementation. This phase provides a reproducible database, secure RPC boundary, TypeScript repository layer, browser ownership-token storage, and automated tests. It does not add sharing UI or GitHub Pages deployment yet.

## Confirmed Product Decisions

- Production frontend will be hosted on GitHub Pages; the Raspberry Pi is a test environment.
- Supabase Free is the initial production backend.
- Posting is anonymous and immediately public. Supabase Auth is not required.
- A browser-generated ownership token authorizes both editing and deletion.
- The database stores only a password hash of that token.
- Losing browser storage permanently loses edit and delete access.
- Owners may replace the complete pack contents and metadata after posting.
- Editing preserves the original creation time and newest-first position while recording a new update time.
- Reports increment a counter. They do not automatically hide a pack.
- Administrators hide packs manually through privileged Supabase access.
- Work proceeds one phase at a time; this phase is backend and repository foundation only.

## Scope

This phase implements:

- a local Supabase project configuration;
- versioned database migrations;
- one private base table and public-result RPC functions;
- database validation, privileges, Row Level Security, and token verification;
- anonymous create, update, soft-delete, report, get, and newest-list operations;
- a frontend-facing `PackRepository` abstraction;
- a localStorage-backed ownership store;
- a Supabase repository implementation and a test double;
- database and TypeScript tests;
- environment and deployment documentation for a new hosted Supabase project and future GitHub Pages builds.

This phase does not implement:

- post, browse, edit, delete, or report UI;
- GitHub Pages workflow files or deployment;
- Supabase Auth;
- automatic moderation;
- strong anti-spam or rate limiting for anonymous reports;
- public filtering by tags, musical key, or author;
- pack version history;
- Edge Functions;
- recommendation ranking from shared-pack statistics.

## Architecture

```text
Static React application
        |
        v
PackRepository + PackOwnershipStore
        |
        v
Supabase RPC over the anon client
        |
        v
Private PostgreSQL table and validation functions
```

React components do not import Supabase or use RPC names. Later sharing UI consumes only `PackRepository`. A different backend or an in-memory test implementation can satisfy the same interface.

The static frontend receives only `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`. A service-role key, database password, personal access token, and ownership-token hashes must never enter the frontend build or committed environment files.

## Database Model

The `public.chord_packs` base table contains:

| Column | Type and rule |
| --- | --- |
| `id` | UUID primary key generated by PostgreSQL |
| `pack_name` | Text, with leading/trailing U+0020 removed, length 1 through 100 Unicode code points |
| `author_name` | Text, with leading/trailing U+0020 removed, length 1 through 50 Unicode code points; blank input becomes `Anonymous` |
| `tags` | Text array, at most 10 entries; each entry has leading/trailing U+0020 removed and is 1 through 30 Unicode code points; PostgreSQL `lower(text)` duplicates are rejected |
| `musical_key` | One of `C`, `C#`, `D`, `D#`, `E`, `F`, `F#`, `G`, `G#`, `A`, `A#`, `B` |
| `track_sound_name` | Text, with leading/trailing U+0020 removed, length 1 through 100 Unicode code points |
| `source_track_index` | Integer `0..9` or null |
| `chords` | JSONB containing exactly seven validated chord slots |
| `reported_count` | Non-negative integer controlled by the server, initially zero |
| `hidden` | Boolean controlled by an administrator, initially false |
| `deleted` | Boolean controlled by owner deletion, initially false |
| `ownership_token_hash` | bcrypt password hash; never returned publicly |
| `created_at` | Server timestamp fixed for the lifetime of the row |
| `updated_at` | Server timestamp changed by a successful owner update or delete |

The table has an index supporting the stable public ordering `created_at DESC, id DESC` and the predicate `hidden = false AND deleted = false`.

### Chord JSON

`chords` is an array of exactly seven objects. Each object contains only:

- `slotIndex`: a unique integer from 1 through 7;
- `notes`: one through four unique integers in the centralized initial relative range `0x24..0x60`;
- `displayName`: a string with leading/trailing U+0020 removed, from 1 through 100 Unicode code points.

Text normalization is defined by PostgreSQL's default `btrim(text)`, which removes U+0020 SPACE only, and text limits use `char_length(text)`, which counts Unicode code points. The TypeScript response boundary applies those same rules; it does not use JavaScript `trim()` or UTF-16 `string.length` for database-normalized fields.

All slot indexes 1 through 7 must be present. The note values remain SEQTRAK-relative; the connected device KEY is not stored or published. JSON validation runs inside create and update RPCs rather than trusting TypeScript validation alone.

### Client and Server Fields

Create and update payloads include only editable fields:

- pack name;
- author name;
- tags;
- musical key;
- track sound name;
- optional source track index;
- seven chords.

Clients cannot set `id`, moderation state, deletion state, report count, ownership hash, or timestamps. Update replaces all editable fields atomically. It preserves `id`, `created_at`, `reported_count`, `hidden`, `deleted`, and the ownership hash, then sets `updated_at` to the server clock.

## RPC Contract

The public Supabase API exposes these PostgreSQL functions:

### `create_pack(payload, ownership_token)`

- Validates the payload and a 64-character hexadecimal ownership token representing 256 random bits.
- Normalizes whitespace and blank author name.
- Hashes the token with bcrypt using `pgcrypto`.
- Inserts one row atomically.
- Returns the public pack representation without a token or token hash.

### `update_pack(pack_id, payload, ownership_token)`

- Verifies the token against the stored bcrypt hash.
- Rejects missing, deleted, or hidden targets and incorrect tokens.
- Validates and replaces all editable fields atomically.
- Preserves creation order and server-owned fields.
- Returns the updated public pack.

### `delete_pack(pack_id, ownership_token)`

- Verifies ownership.
- Sets `deleted = true` and updates `updated_at`.
- Does not physically remove the row.
- Returns no public pack.

### `report_pack(pack_id)`

- Operates only on a currently public pack.
- Atomically increments `reported_count` by one.
- Does not change `hidden`.
- Anonymous repeat reports remain possible in this phase; the count is an administrative signal, not proof of unique reporters.

### `get_pack(pack_id)`

- Returns one public pack when `hidden = false AND deleted = false`.
- Returns no row for missing, hidden, or deleted packs.

### `list_packs(limit, cursor_created_at, cursor_id)`

- Returns public packs ordered by `created_at DESC, id DESC`.
- Uses a `(created_at, id)` cursor rather than an offset so inserts do not reorder an in-progress traversal.
- Defaults to 20 rows and accepts limits from 1 through 100.
- Returns the next cursor when another page may exist.
- Does not add tag, key, or author filters in this phase.

Every public pack result explicitly selects safe columns. The base table row type and `ownership_token_hash` are never returned.

## Ownership Token Lifecycle

`SupabasePackRepository.createPack` generates 32 random bytes with `crypto.getRandomValues()` and serializes them as 64 lowercase hexadecimal characters. The token is sent once to `create_pack`. After the RPC response passes strict public-pack validation, the repository stores `{ packId, token }` through `PackOwnershipStore`.

The default localStorage implementation uses a versioned application key and treats malformed stored JSON as empty ownership state. It supports:

```ts
interface PackOwnershipStore {
  save(packId: string, token: string): void;
  get(packId: string): string | null;
  remove(packId: string): void;
}
```

Update and delete obtain the token through this interface. Missing local ownership fails before making a Supabase request. Successful deletion removes the local token. A failed operation keeps it so transient backend errors remain retryable.

Tokens are never logged, shown in ordinary errors, included in URLs, returned by list/get, or persisted inside `ChordPack`.

## Database Security

- Enable RLS on `public.chord_packs`.
- Do not grant `anon` or `authenticated` direct insert, update, or delete privileges on the base table.
- Do not provide a base-table select path that exposes `ownership_token_hash`.
- Revoke default function execution from `PUBLIC`, then grant only the intended RPC functions to `anon` and `authenticated`.
- Implement public RPCs as `SECURITY DEFINER SET search_path = ''` and fully qualify every database object.
- Install `pgcrypto` in the `extensions` schema and fully qualify bcrypt functions.
- Validate payloads and ownership before mutation.
- Give client errors no secret hash or database-internal details.
- Keep administrator moderation outside the anonymous RPC surface; service-role or direct privileged database access may set `hidden`.

Supabase recommends an empty search path with schema-qualified references for security-definer functions. The design follows that pattern: [Supabase database functions](https://supabase.com/docs/guides/database/functions).

## Frontend Repository Boundary

The application-facing contract is conceptually:

```ts
interface PackRepository {
  createPack(pack: EditablePack): Promise<PublicPack>;
  updatePack(pack: PublicPack): Promise<PublicPack>;
  deletePack(packId: string): Promise<void>;
  reportPack(packId: string): Promise<void>;
  getPack(packId: string): Promise<PublicPack | null>;
  listPacks(options?: ListPackOptions): Promise<PackPage>;
}
```

The implementation separates:

- public DTO and RPC payload mapping;
- Supabase client creation from environment configuration;
- ownership-token generation and persistence;
- RPC error mapping;
- an in-memory or mocked repository for UI tests.

The existing editor continues to use `ChordPack`. Mapping code strips server-controlled fields when creating or updating and restores them from public responses. The device-wide SEQTRAK KEY remains transient and is not added to shared data.

## Error Model

Repository callers receive stable application errors for:

- missing Supabase URL or anon key;
- invalid editable pack data;
- missing local ownership;
- rejected ownership token;
- missing, hidden, or deleted pack;
- network or Supabase service failure;
- malformed or incompatible RPC response.

Messages do not expose tokens, hashes, SQL text, or privileged metadata. A wrong token and a target that cannot be owner-mutated may share the same ownership rejection so the RPC does not reveal unnecessary state. Retriable transport failures remain distinguishable from validation and ownership failures.

## Local and Hosted Workflow

The repository commits `supabase/config.toml`, migrations, and pgTAP tests. CLI scratch state and all secrets remain ignored.

Local workflow:

```text
npx supabase start
npx supabase db reset
npx supabase test db
```

The local stack is development-only and must not be exposed. Current Supabase documentation requires a Docker-compatible runtime and recommends at least 7 GB of memory for the CLI stack. If the Raspberry Pi cannot provide that capacity, the same committed setup may run on another development machine without changing the schema or tests: [Supabase local workflow](https://supabase.com/docs/guides/local-development/cli-workflows).

Hosted deployment workflow:

1. User creates a new Supabase Free project.
2. Authenticate the CLI with a personal access token.
3. Link the repository to the new project.
4. Preview pending migrations.
5. Push migrations once.
6. Configure `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` in the later GitHub Pages build.

```text
npx supabase login
npx supabase link --project-ref <project-ref>
npx supabase db push --dry-run
npx supabase db push
```

Remote schema changes must be made through committed migrations rather than directly in the production dashboard. This follows Supabase's migration guidance: [Supabase database migrations](https://supabase.com/docs/guides/deployment/database-migrations).

## Testing

### Database Tests

pgTAP tests under `supabase/tests` run through `npx supabase test db`. Supabase executes each database test in a transaction. Tests cover:

- expected table, columns, constraints, RLS, privileges, and functions;
- valid anonymous creation and public retrieval;
- invalid metadata, tag, slot, and note payload rejection;
- omission of ownership hashes and server-only fields from results;
- correct-token full update;
- preservation of `created_at`, report count, and moderation fields on update;
- wrong-token update and delete rejection;
- soft deletion and removal from get/list;
- report increment on a public pack;
- report rejection for hidden or deleted packs;
- public list exclusion of hidden and deleted rows;
- stable cursor ordering and page boundaries;
- inability of anonymous roles to mutate the base table directly.

This uses the officially supported pgTAP workflow: [Supabase database testing](https://supabase.com/docs/guides/local-development/testing/overview).

### TypeScript Tests

Vitest tests cover:

- 256-bit lowercase hexadecimal token generation;
- ownership save, read, overwrite, removal, and corrupt-storage recovery;
- saving ownership only after a successful create;
- selecting the stored token for update and delete;
- removing ownership only after a successful delete;
- mapping between `ChordPack`, editable payloads, and public responses;
- mapping Supabase failures to stable application errors;
- repository substitution without React or live Supabase;
- preservation of existing local editor, MIDI, KEY-relative, and device tests.

## Completion Criteria

The backend-foundation phase is complete when:

- `npx supabase db reset` succeeds from an empty local database;
- all pgTAP database tests pass;
- all Vitest tests, static-server tests, and the production build pass;
- migrations can be dry-run and pushed to the new hosted Supabase project;
- the hosted project works using only the public anon key for the supported RPCs;
- create, get/list, owner update, owner soft-delete, and report are verified;
- ownership hashes and privileged credentials are absent from public results and frontend build inputs;
- setup, testing, linking, deployment, environment, and recovery steps are documented;
- no sharing UI or GitHub Pages deployment behavior is coupled into the repository layer.
